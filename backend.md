## 一、操作系统

### Q1: 进程、线程、协程有什么区别？各自的适用场景是什么？

**题目解析**：并发基础知识，是所有后端岗位的必考题，理解三者有助于做正确的并发模型选择。

**题目讲解**：
**进程（Process）**：
- 操作系统资源分配的基本单位，有独立的虚拟地址空间、文件描述符等
- 进程间通信（IPC）：管道、消息队列、共享内存、Socket、信号
- 切换代价：需保存整个进程上下文，代价高（上下文切换 μs 级）
- 隔离性好，一个进程崩溃不影响其他进程

**线程（Thread）**：
- CPU 调度的基本单位，同进程内的线程共享地址空间和文件描述符
- 线程间通信：共享内存（需要同步），比进程通信快
- 切换代价：比进程小，但仍需切换寄存器、栈等
- 并发问题：多线程共享数据需要加锁（互斥量、读写锁等），易产生死锁、竞态条件

**协程（Coroutine）**：
- 用户态轻量线程，由用户代码调度（非 OS），一个线程可以跑数千个协程
- 切换代价：极低，只需保存/恢复几个寄存器，不涉及系统调用
- 协作式调度：协程主动让出（yield/await），非抢占式
- Python asyncio、Go goroutine、Lua coroutine

**选择依据**：
- CPU 密集型：多进程（Python 绕过 GIL）
- I/O 密集型、高并发网络：协程（asyncio / Go）
- 后台任务并行：线程池

**考察点**：
1. 进程 vs 线程的地址空间区别
2. 协程和线程的调度方式区别（OS 调度 vs 用户调度）
3. Python GIL 对多线程的影响

**面试官更想听**：
结合 Python 说出 GIL 使得多线程无法并行执行 CPU 密集型任务，应用场景要用 multiprocessing；I/O 密集型用 asyncio 协程效率更高。

**示例答案**：
进程是资源分配的基本单位，有独立内存空间，隔离性强，进程崩溃不影响他人，但创建和切换代价大。线程共享进程的内存，切换代价比进程低，适合并行处理但要注意线程安全（加锁）。协程是用户态的并发，由代码控制调度（yield/await），切换代价极低，适合 I/O 密集、高并发场景——Go 的 goroutine 可以轻松启动百万个，asyncio 的协程也比线程省资源得多。Python 里因为 GIL，多线程不能真正并行执行 Python 字节码（CPU 密集型任务还是串行），CPU 密集任务要用 multiprocessing 绕过 GIL；I/O 密集型（网络请求、数据库查询）用 asyncio 协程，切换代价极小且不需要 GIL，性能远超多线程。

### Q2: 什么是死锁？死锁的四个必要条件和预防策略是什么？

**题目解析**：死锁是多线程编程的核心安全问题，后端面试必考。

**题目讲解**：
**死锁定义**：两个或多个线程因互相持有对方需要的资源，无限期等待的状态。

**四个必要条件（Coffman 条件）**：
1. **互斥（Mutual Exclusion）**：资源同一时刻只能被一个线程持有
2. **持有并等待（Hold and Wait）**：线程持有资源同时等待其他资源
3. **不可剥夺（No Preemption）**：资源不能被强制剥夺，只能主动释放
4. **循环等待（Circular Wait）**：存在 T1→T2→...→Tn→T1 的循环等待链

**预防策略**（破坏任一条件）：
1. 破坏持有并等待：一次性申请所有资源（或申请失败全部释放）
2. 破坏不可剥夺：允许剥夺（优先级抢占，或 tryLock + 超时）
3. 破坏循环等待：对资源编号，按固定顺序申请（最常用）

**检测与恢复**：
- 资源分配图（Resource Allocation Graph）检测环路
- 恢复：杀死进程、回滚事务、强制剥夺资源

**Python 死锁示例**：
```python
import threading
lock1 = threading.Lock()
lock2 = threading.Lock()
# 线程1: lock1 → lock2，线程2: lock2 → lock1 → 死锁
```

**考察点**：
1. 四个必要条件（缺一不构成死锁）
2. 加锁顺序统一是最实用的预防手段
3. 数据库死锁（两个事务互相等待行锁）

**示例答案**：
死锁发生需要同时满足四个条件：资源互斥、持有并等待、不可剥夺、循环等待。只要破坏其中一个就能预防死锁。最常用的是破坏循环等待：规定所有线程必须按资源编号从小到大申请锁，这样不可能形成环路。Python 里 `threading.RLock` 支持可重入，`lock.acquire(timeout=1)` 支持超时放弃（破坏不可剥夺）。数据库里死锁是事务级别的——两个事务互相持有对方需要的行锁，MySQL InnoDB 有死锁检测机制，发现死锁后自动回滚代价较小的那个事务，业务代码要做好重试逻辑。在 Redis 分布式锁里，加锁时设置过期时间（TTL）可以防止持锁进程崩溃导致的"永久死锁"。

### Q3: IO 模型有哪几种？epoll 为什么比 select/poll 高效？

**题目解析**：IO 模型是理解高性能服务器（Nginx、Redis）的基础，后端面试必考。

**题目讲解**：
**五种 IO 模型**：
1. **阻塞 IO（BIO）**：调用 read()，阻塞直到数据可读
2. **非阻塞 IO（NIO）**：调用 read() 立即返回，若无数据返回 EAGAIN，需要轮询
3. **IO 多路复用**：select/poll/epoll，一个线程监控多个 fd，有就绪时通知
4. **信号驱动 IO（SIGIO）**：数据就绪时操作系统发送 SIGIO 信号（少用）
5. **异步 IO（AIO）**：注册回调，内核完成数据拷贝后回调（Windows IOCP 成熟，Linux AIO 少用）

**select vs poll vs epoll**：

| 特性 | select | poll | epoll |
|------|--------|------|-------|
| 最大 fd | 1024（FD_SETSIZE）| 无限制 | 无限制 |
| 就绪检测 | 遍历全部 fd（O(N)）| 遍历全部 fd（O(N)）| 事件通知（O(1)）|
| 内存拷贝 | 每次调用都要用户→内核拷贝 fd 集合 | 同 select | fd 注册一次，事件驱动 |
| 内核实现 | 位图 | pollfd 数组 | 红黑树 + 就绪链表 |

**epoll 工作模式**：
- **LT（水平触发）**：只要 fd 可读就一直通知，类似 select
- **ET（边缘触发）**：只在状态变化时通知一次，需要配合非阻塞 IO 一次读完

**考察点**：
1. epoll 的数据结构（红黑树存 fd，就绪链表存事件）
2. ET 模式为什么需要非阻塞 IO
3. Nginx/Redis 如何利用 epoll 实现高并发

**示例答案**：
select 和 poll 每次调用都需要把所有监控的 fd 从用户空间拷贝到内核，内核遍历全部 fd 检查就绪状态（O(N)），N 大了性能很差，select 还有 1024 的 fd 上限。epoll 用红黑树在内核维护所有注册的 fd（注册一次，无需每次拷贝），用就绪链表存储已就绪的 fd，每次 epoll_wait 只返回就绪的 fd（O(1) 获取事件），大幅提升了高并发场景的效率。Nginx 和 Redis 都用 epoll 实现事件驱动，单线程/单进程就能处理数万并发连接。epoll 的 ET 模式效率更高但编程难度大：状态变化时只通知一次，必须配合非阻塞 IO 循环读取到 EAGAIN，否则会漏数据。

### Q4: MySQL 的 B+ 树索引是怎么工作的？为什么不用 B 树或哈希索引？

**题目解析**：索引是 MySQL 性能优化的核心，理解 B+ 树是理解所有索引行为的基础。

**题目讲解**：
**B+ 树结构**：
- 多路平衡搜索树，所有数据都存在叶子节点
- 叶子节点通过链表连接（范围查询高效）
- 内部节点只存键值，不存数据（每个节点能存更多键，树更矮）
- 查询、插入、删除均 O(log N)

**为什么不用 B 树**：
- B 树内部节点也存数据，节点能存的键少，树高更高，IO 次数更多
- B 树不支持高效的范围查询（需要中序遍历，节点可能不连续）

**为什么不用哈希索引**：
- 哈希索引只支持等值查询（`=`），不支持范围查询（`>`、`<`、`BETWEEN`）
- 不支持排序（ORDER BY）
- 哈希冲突严重时性能下降
- 不支持最左前缀匹配

**B+ 树索引的特性**：
- **聚簇索引（主键索引）**：叶子节点存完整行数据，每个表只有一个
- **二级索引（辅助索引）**：叶子节点存主键值，查完后需要"回表"查聚簇索引
- **最左前缀匹配**：联合索引 (a,b,c)，查询必须从 a 开始匹配
- **覆盖索引**：查询所需字段全在索引中，无需回表

**考察点**：
1. 聚簇索引 vs 二级索引的存储区别
2. 回表的含义和优化（覆盖索引）
3. 联合索引的最左前缀原则

**示例答案**：
B+ 树是 MySQL InnoDB 的核心数据结构。与 B 树的区别是：B+ 树内部节点只存键，所有数据在叶子节点，叶子节点形成有序链表。这样每个内部节点能存更多键（树更矮），范围查询直接在叶子链表遍历（不用回溯）。相比哈希索引，B+ 树支持范围查询、排序、最左前缀匹配。InnoDB 的聚簇索引以主键为 B+ 树的键，叶子节点存完整行数据；二级索引叶子节点存主键值，查询时需要"回表"用主键到聚簇索引再查一次完整数据。覆盖索引优化回表：如果查询的字段全包含在索引里（如 `SELECT id, name FROM user WHERE name='Alice'`，有 `(name, id)` 索引），直接在二级索引完成查询，避免回表。联合索引 `(a,b,c)` 遵循最左前缀：`WHERE a=1 AND b=2` 能用到索引，`WHERE b=2` 则用不到（必须从最左列 a 开始）。

### Q5: MySQL 事务的四个隔离级别分别是什么？MVCC 是如何工作的？

**题目解析**：事务隔离是数据库面试的核心，MVCC 是 MySQL 实现高并发读的关键机制。

**题目讲解**：
**并发问题**：
- **脏读**：读到其他事务未提交的数据（提交前可能回滚）
- **不可重复读**：同一事务两次读同一行，值不同（中间被其他事务 UPDATE）
- **幻读**：同一事务两次范围查询，结果集不同（中间被其他事务 INSERT/DELETE）

**四个隔离级别**：
| 级别 | 脏读 | 不可重复读 | 幻读 |
|------|------|-----------|------|
| READ UNCOMMITTED | 可能 | 可能 | 可能 |
| READ COMMITTED | ✅ | 可能 | 可能 |
| REPEATABLE READ（MySQL 默认）| ✅ | ✅ | ✅（InnoDB MVCC+间隙锁防止）|
| SERIALIZABLE | ✅ | ✅ | ✅ |

**MVCC（多版本并发控制）**：
- 每行数据有两个隐藏字段：`trx_id`（最后修改该行的事务 ID）和 `roll_pointer`（指向 undo log 中旧版本）
- Read View：事务开始时创建快照，记录当前活跃事务 ID 列表
- 读取时，通过 `trx_id` 和 Read View 判断版本是否可见：
  - 版本的 trx_id < min_trx_id（ReadView 创建时最小活跃事务 ID）→ 可见
  - 版本的 trx_id > max_trx_id → 不可见，通过 roll_pointer 查旧版本
- **RC 和 RR 的区别**：RC 每次 SELECT 创建新 Read View，RR 只在事务开始时创建一次

**考察点**：
1. 幻读 vs 不可重复读的区别（行 vs 范围）
2. RR 级别下 MVCC 防止不可重复读的机制
3. 当前读 vs 快照读（SELECT FOR UPDATE 是当前读，不走 MVCC）

**示例答案**：
MySQL 默认的隔离级别是 REPEATABLE READ，通过 MVCC 实现了无锁的一致性读。每行数据在 undo log 里保存历史版本，事务开始时创建 Read View（快照），记录当前所有活跃事务。读取时，通过版本链和 Read View 找到"对当前事务可见"的最新版本——比当前事务早提交的版本可见，当前事务创建后才开始的事务修改的版本不可见。RR 级别的 Read View 在整个事务期间只创建一次，所以两次读到的数据一致（防不可重复读）；RC 级别每次 SELECT 都创建新 Read View，能看到最新提交，所以会出现不可重复读。InnoDB 在 RR 级别还通过间隙锁（Gap Lock）解决了范围查询的幻读：`SELECT ... WHERE id BETWEEN 1 AND 10 FOR UPDATE` 会锁住 1-10 的间隙，防止其他事务插入新行。注意：FOR UPDATE 是当前读（读最新版本+加锁），不走 MVCC。

### Q6: 如何优化慢查询？EXPLAIN 的关键字段有哪些？

**题目解析**：SQL 优化是后端日常工作，也是面试必考的实战题。

**题目讲解**：
**慢查询优化步骤**：
1. 开启慢查询日志（`slow_query_log=ON`，`long_query_time=1`）
2. 找出慢查询语句
3. `EXPLAIN` 分析执行计划
4. 针对性优化

**EXPLAIN 关键字段**：
- **type**（访问类型，从优到劣）：system > const > eq_ref > ref > range > index > ALL
  - `ALL`：全表扫描（最差，务必优化）
  - `range`：范围扫描（索引范围查询，可接受）
  - `ref`：非唯一索引等值查询
  - `const`：主键等值查询（最快）
- **key**：实际使用的索引（NULL 表示没用索引）
- **key_len**：使用的索引长度（联合索引判断用了哪些列）
- **rows**：预估扫描行数（越小越好）
- **Extra**：
  - `Using index`：覆盖索引（好）
  - `Using filesort`：需要额外排序（坏，应用索引覆盖 ORDER BY）
  - `Using temporary`：用了临时表（坏）

**常见优化手段**：
1. 为 WHERE/JOIN/ORDER BY 字段添加合适索引
2. 避免在索引列上做函数运算（`WHERE YEAR(created_at) = 2024` 索引失效）
3. 覆盖索引减少回表
4. 分页优化（`LIMIT 100000, 10` 很慢，用 cursor 分页 `WHERE id > last_id LIMIT 10`）
5. JOIN 小表驱动大表（小表作为 Nested Loop 的外层）

**考察点**：
1. EXPLAIN type 各值的含义和触发条件
2. 索引失效的常见场景（函数、隐式转换、like '%abc'）
3. 大数据量分页的 cursor 优化

**示例答案**：
慢查询优化的标准流程：先从慢查询日志或监控平台找出耗时 SQL，EXPLAIN 分析执行计划，重点看 type（是否全表扫描）、key（是否走索引）、rows（扫描行数）、Extra（是否 filesort）。type=ALL 说明全表扫描，必须优化；rows 过大说明索引选择性差或没有索引。加索引时注意：索引列上有函数调用（`WHERE DATE(created_at)='2024-01-01'`）会导致索引失效，应改写为 `WHERE created_at >= '2024-01-01' AND created_at < '2024-01-02'`。LIKE `'%key%'` 前缀模糊查询走不了 B+ 树，可以考虑全文索引或 Elasticsearch。大分页 `LIMIT offset, n` 的问题是 MySQL 需要扫描并丢弃前 offset 行，用 cursor 分页（记录上次最大 id，下次 `WHERE id > last_id LIMIT n`）可以把 O(N) 降为 O(1)，效果显著。联合索引要注意最左前缀原则，(a,b,c) 的索引只有 WHERE 从 a 开始才生效。

### Q7: Redis 的六种数据类型和底层数据结构是什么？

**题目解析**：Redis 是后端必用中间件，理解其数据结构有助于选择正确的类型和理解性能特征。

**题目讲解**：
| 数据类型 | 底层实现 | 适用场景 |
|---------|---------|---------|
| String | SDS（简单动态字符串）| 缓存、计数器、分布式锁 |
| List | quicklist（ziplist + 双向链表）| 消息队列、最新列表（LPUSH+LRANGE）|
| Hash | ziplist（小）/ hashtable（大）| 对象存储（用户信息）|
| Set | intset（整数集合）/ hashtable | 标签、去重、交并差集 |
| Sorted Set | ziplist（小）/ skiplist + hashtable | 排行榜、延迟队列 |
| Stream | 专用 listpack + Radix Tree | 消息队列（支持 consumer group）|

**跳表（Skip List）**：
- 有序链表的多层索引，查询 O(log N)
- 为什么 ZSet 用跳表而非红黑树：
  - 跳表范围查询（ZRANGEBYSCORE）只需找到起始节点沿链表遍历，红黑树需要中序遍历更复杂
  - 跳表实现简单，内存占用接近
  - 支持并发更新性能好

**SDS（Simple Dynamic String）vs C 字符串**：
- 记录字符串长度（O(1) 获取），C 字符串 O(N)
- 预分配空间，减少内存重分配
- 二进制安全（可存 `\0`）

**考察点**：
1. 跳表的时间复杂度和使用场景
2. ziplist 何时转为其他结构（元素数量阈值）
3. Redis 5.0 新增 Stream 的优势

**示例答案**：
Redis 六种数据类型各有不同的底层实现。String 底层是 SDS（自研动态字符串），相比 C 字符串有 O(1) 的 len 字段、自动扩容、二进制安全。List 是 quicklist，本质是分段的 ziplist + 双向链表，小 list 用紧凑的 ziplist，大了拆分成链表节点。Hash 小时用 ziplist 节省内存，超过阈值（默认 512 字段或 64 字节）转 hashtable。Sorted Set 用跳表 + hashtable，跳表保证有序，hashtable 保证 score 查询 O(1)，相比红黑树，跳表在范围查询时直接链表遍历更自然，代码也更简单。Set 纯整数时用 intset（有序整数数组），效率很高，否则用 hashtable。Stream 是 Redis 5.0 加的，支持消费者组（consumer group）、ACK 机制，适合需要可靠性保证的消息队列场景，比 List 更像专业 MQ。

### Q8: Redis 如何实现分布式锁？Redlock 算法解决了什么问题？

**题目解析**：分布式锁是微服务架构的核心问题，面试官几乎必问，且会深挖细节。

**题目讲解**：
**基础实现（单 Redis 节点）**：
```bash
SET lock_key unique_value NX PX 30000
# NX: 不存在才设置（原子性）, PX: 毫秒级过期时间
```

**释放锁（Lua 脚本保证原子性）**：
```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```
用 unique_value（UUID）标识锁的持有者，防止误删他人的锁。

**单节点问题**：Redis 主从切换时，主节点加锁后若未同步到从节点就宕机，新主节点上没有该锁，其他客户端可以再次加锁，产生并发问题。

**Redlock 算法（多节点）**：
- 对 N（通常 5）个独立 Redis 节点依次加锁
- 超过 N/2+1 个节点加锁成功，且总耗时 < 锁过期时间，才算加锁成功
- 失败时释放所有已加的锁
- 争议：Martin Kleppmann 认为仍有问题（GC pause 导致锁失效）

**工程实践**：
- Java 用 Redisson（实现了 Redlock + 看门狗续期）
- Python 用 redis-py 手写或 pottery 库
- 锁续期（Watchdog）：后台线程定期延长锁 TTL，防止任务未完成锁过期

**考察点**：
1. SET NX PX 的原子性
2. Lua 脚本防止误删的原理
3. Redlock 的多数票原则

**示例答案**：
Redis 分布式锁基础实现：`SET lock:key my_uuid NX PX 30000`，NX 保证只有一个客户端设置成功（原子操作），PX 设置过期时间防止死锁（持锁进程崩溃后自动释放）。释放时要用 Lua 脚本：先检查 value 是否是自己的 UUID，是才删除，防止锁过期后误删别人的锁（GET+DEL 非原子，Lua 保证原子性）。单节点的问题是主从切换时锁可能丢失，Redlock 用 5 个独立 Redis 节点，需要超过半数（3个）成功才算加锁，某一个节点宕机不影响整体。Redisson 是 Java 里最成熟的实现，自带看门狗（Watchdog）机制：持锁期间后台线程每 10 秒检查是否还持有锁，若是则续期 30 秒，防止业务执行时间超过锁 TTL 导致锁提前失效。Python 项目里通常用 `redis-py` 手写 Lua 脚本实现，或用 `pottery` 库的 RedisSimpleLock。

### Q9: Redis 的缓存穿透、缓存击穿、缓存雪崩分别是什么？如何解决？

**题目解析**：缓存高可用三大问题，是后端面试的高频考察点，需要熟练说出每种情况的解决方案。

**题目讲解**：

**缓存穿透（Cache Penetration）**：
- **现象**：查询不存在的 key（如 id=-1），缓存和数据库都没有，每次请求都打到 DB
- **原因**：恶意攻击或业务异常
- **解决**：
  1. **布隆过滤器**：预先把所有合法 key 放入 Bloom Filter，请求到来先判断 key 是否可能存在，不存在直接拒绝
  2. **空值缓存**：查 DB 不存在时，缓存空值（`""` 或 `null`）并设短 TTL

**缓存击穿（Cache Breakdown）**：
- **现象**：热点 key 恰好过期，大量并发请求同时打到 DB
- **解决**：
  1. **互斥锁**：缓存 miss 时先加锁，只有一个线程查 DB 并回填缓存，其他线程等待重试
  2. **逻辑过期**：缓存不设物理过期时间，value 里存逻辑过期时间，过期后异步更新缓存

**缓存雪崩（Cache Avalanche）**：
- **现象**：大量 key 同时过期，或 Redis 宕机，大量请求打到 DB
- **解决**：
  1. **随机过期时间**：在基础 TTL 上加随机偏移，避免同一时刻大量 key 过期
  2. **Redis 高可用**：主从复制 + 哨兵或 Redis Cluster
  3. **限流 + 降级**：请求 DB 的速率限流，压不垮的情况下降级返回兜底数据

**考察点**：
1. 三者的区别（不存在的 key / 热点 key 过期 / 大量 key 过期）
2. 布隆过滤器的假阳性问题
3. 互斥锁实现的细节（setnx + expiry）

**示例答案**：
三个缓存问题的区分：穿透是查询压根不存在的数据（DB 也没有），靠布隆过滤器拦截或缓存空值；击穿是热点 key 在高并发瞬间过期，靠加互斥锁或逻辑过期解决；雪崩是大批 key 同时过期（或 Redis 宕机），靠随机 TTL + Redis 高可用解决。布隆过滤器的注意点：有假阳性（可能把合法 key 判断为不存在），误判率与位数组大小和哈希函数数量有关，Redis 4.0 引入了 RedisBloom 模块可以直接用。互斥锁防击穿：`SET lock NX EX 5`，拿到锁的线程查 DB 回填缓存，没拿到的 sleep(50ms) 后重试（通常 1-2 次就能命中刚回填的缓存）。逻辑过期方案：缓存永不过期（no TTL），value 里附带 expire_time，读到后台异步更新，同时返回旧数据，牺牲一致性换取更高可用性，适合能接受短暂旧数据的场景。
