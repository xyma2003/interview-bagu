### Q58: 竞态条件是什么？互斥锁、信号量、条件变量分别解决什么问题？

**题目解析**：并发 bug 的根源和同步原语的精确区分，是多线程编程的基础，大厂面试必考。

**题目讲解**：

**竞态条件（Race Condition）**：

多个线程并发访问共享资源，最终结果取决于线程执行的时序，属于不确定行为。根本原因是非原子操作被打断后出现指令交错（Instruction Interleaving）：

```python
# counter += 1 在 CPU 层面分解为三步：
# 1. LOAD  counter → 寄存器
# 2. 寄存器 +1
# 3. STORE 寄存器 → counter

# Thread A         Thread B
# LOAD (=0)
#                  LOAD (=0)
# STORE (=1)
#                  STORE (=1)   ← 期望 2，实际 1，数据丢失
```

**三大同步原语对比**：

| 原语 | 本质 | 解决的问题 |
|------|------|-----------|
| 互斥锁（Mutex） | 0/1 二值信号，保证临界区互斥 | 共享资源的互斥访问 |
| 信号量（Semaphore） | 计数器，允许 N 个并发访问 | 资源池限流、生产者-消费者同步 |
| 条件变量（Condition） | 等待某条件成立后被唤醒 | 线程间"等待状态变化"的协调 |

**互斥锁用法**：

```python
import threading
lock = threading.Lock()
counter = 0

def increment():
    global counter
    with lock:          # 同一时刻只有一个线程能进入临界区
        counter += 1
```

**信号量用法**（限制最多 3 个线程同时访问数据库）：

```python
sem = threading.Semaphore(3)

def query_db():
    with sem:            # 超过 3 个线程时，其余阻塞等待
        db.execute(...)
```

> 类比：停车场有 3 个车位，满了在外面等，有车出来才能进。

**条件变量用法**（生产者-消费者模型）：

```python
cond = threading.Condition(lock)

# 消费者
with cond:
    while queue.empty():
        cond.wait()        # 原子地释放锁 + 进入睡眠
    item = queue.get()

# 生产者
with cond:
    queue.put(item)
    cond.notify()          # 唤醒一个等待线程
```

注意：`while` 而非 `if`，是因为 **虚假唤醒（Spurious Wakeup）**——被唤醒后要重新检查条件。

**锁的粒度原则**：
- 锁的粒度越细并发度越高，但越复杂越容易出 bug
- 持锁时间越短越好，I/O 操作、长计算不应在锁内进行

**考察点**：
1. 竞态条件和指令交错的关系（非原子操作被打断）
2. 三种原语的本质差别和选型依据
3. 条件变量必须搭配 while 检查的原因（虚假唤醒）

**示例答案**：
竞态条件的本质是多个线程以不确定的顺序操作共享数据，`counter += 1` 看起来是一行代码，但在 CPU 层面是"读-改-写"三步，两个线程可以在任意步骤之间切换，导致一次 +1 被覆盖。解决竞态必须把多步操作变成"临界区"，保证原子性。互斥锁（Mutex）是最基础的原语，同一时刻只有一个线程能进入临界区，适合保护任意共享资源。信号量（Semaphore）是 Mutex 的泛化，允许 N 个线程并发进入，适合限制对资源池（连接池、线程池）的并发访问。条件变量必须搭配锁使用，适合"等待某个状态变化"的场景，比如生产者-消费者：消费者发现队列空了就 `wait()`（原子释放锁并睡眠），生产者放入元素后 `notify()` 唤醒。必须用 `while` 循环检查条件而非 `if`，因为操作系统可能产生虚假唤醒（被唤醒了但条件没变），`while` 确保唤醒后一定重新确认条件成立才继续。

### Q59: 什么是无锁编程？CAS 操作的原理和 ABA 问题如何解决？

**题目解析**：无锁编程是高性能并发的核心技术，CAS 是 Java `AtomicInteger`、Python `multiprocessing` 等底层实现的基础，面试考察候选人对并发底层的理解深度。

**题目讲解**：

**为什么要无锁？**

互斥锁的代价：
- 抢不到锁的线程被挂起 → 上下文切换开销（μs 级）
- 锁持有者崩溃 → 其他线程永久阻塞
- 高竞争时串行化严重，扩展性差

**CAS（Compare-And-Swap）原理**：

CPU 提供的原子指令，操作如下：

```
CAS(address, expected, new_value):
    if *address == expected:   # 比较：当前值是否是预期值
        *address = new_value   # 交换：更新为新值
        return True
    else:
        return False           # 有人改过了，失败
// 以上是不可分割的原子操作
```

**CAS 实现无锁自增**：

```python
# 伪代码
while True:
    old = atomic_read(counter)
    new = old + 1
    if CAS(counter, old, new):   # 没人在我读和写之间修改 → 成功
        break
    # 有人抢先修改了 → 重试（自旋）
```

这种模式叫**自旋 + CAS（Spin-CAS）**，失败则不断重试，适合竞争不激烈的场景。

**ABA 问题**：

```
Thread A: 读到 counter = A
Thread B: 改 counter: A → B → A（改来改去又回到 A）
Thread A: CAS(counter, A, new) → 成功！但中间发生了改变

在某些数据结构里，值"回到 A"不代表状态没变
例如：链表节点被移除后又重新加入，ABA 导致操作的是"已删除"的节点
```

**ABA 解决方案：加版本号（Stamped Reference）**

```java
// Java AtomicStampedReference
AtomicStampedReference<Integer> ref = new AtomicStampedReference<>(100, 0);

int[] stamp = new int[1];
int value = ref.get(stamp);     // 同时获取值和版本号

// CAS 必须值和版本号都匹配才成功
ref.compareAndSet(value, newValue, stamp[0], stamp[0] + 1);
// 即使值回到 100，版本号已经是 2 了，CAS 失败
```

**无锁数据结构**：

| 数据结构 | 实现方式 |
|---------|---------|
| 无锁栈 | CAS 替换栈顶指针 |
| 无锁队列 | Michael-Scott 队列（CAS 维护 head/tail） |
| 无锁哈希表 | CAS 插入链表节点 |

**考察点**：
1. CAS 是 CPU 原子指令，不需要操作系统参与（不阻塞线程）
2. ABA 的危害和版本号方案（AtomicStampedReference）
3. CAS 适合低竞争，高竞争下自旋会消耗大量 CPU

**示例答案**：
CAS 是 CPU 层面的"先比较再交换"原子操作：读当前值、和预期值比较，相同才写入新值，整个过程不可分割，不需要加锁。Java 的 `AtomicInteger.incrementAndGet()` 底层就是 `CAS(addr, old, old+1)` 的自旋循环。CAS 的优势是不阻塞线程，失败只是重试（自旋），适合竞争少的场景；高竞争时大量线程自旋会浪费 CPU，这时候加锁反而更好（JVM 有自适应自旋，自旋超时后升级为阻塞）。ABA 问题：如果内存值从 A 变成 B 再变回 A，CAS 以为什么都没发生，但中间的状态变化是真实的，在链表等数据结构里可能操作已经失效的节点。解决方案是加版本号：Java 的 `AtomicStampedReference` 把"值+版本号"一起参与 CAS，即使值回到 A，版本号已经不同了，CAS 失败，避免了 ABA 问题。Python 的 `multiprocessing.Value` 通过锁实现原子操作，C 扩展代码可以直接用 `__atomic_compare_exchange_n` 等 GCC 内置函数。

### Q60: 分布式内存模型：CPU 缓存一致性和内存屏障是什么？volatile 和 atomic 的区别？

**题目解析**：内存可见性是多线程编程中最容易被忽视的底层问题，大厂面试考察候选人对 CPU 体系结构的理解深度。

**题目讲解**：

**问题根源：CPU 多级缓存**

```
CPU Core 0              CPU Core 1
    ↓                       ↓
 L1 Cache               L1 Cache
    ↓                       ↓
 L2 Cache               L2 Cache
    ↓                       ↓
         L3 Cache（共享）
              ↓
           主内存（RAM）
```

- 每个核心有独立 L1/L2 缓存，写操作可能只在本核心缓存里
- 另一个核心从主内存或自己的缓存读，可能读到旧值
- 此外，编译器和 CPU 会对指令重排（Reordering）优化性能，破坏多线程的执行顺序预期

**可见性问题示例**：

```python
# Python 中由于 GIL 通常不会出现此问题
# 但在 Java/C++ 中很典型
flag = False

# Thread A
while not flag:    # 可能永远循环：flag 的修改在 B 的缓存里没刷到主内存
    pass

# Thread B
flag = True        # 写到了 B 的 L1 Cache，但 A 不可见
```

**内存屏障（Memory Barrier / Memory Fence）**：

CPU 提供的特殊指令，强制两件事：
1. **刷缓存**：确保屏障前的写操作对其他核心可见
2. **禁止重排**：屏障前的操作不会被排到屏障后

```
StoreStore Barrier：保证屏障前的写不会被重排到屏障后的写之后
LoadLoad Barrier：保证屏障前的读不会被重排到屏障后的读之后
StoreLoad Barrier：最强，全能屏障（代价最大）
```

**Java `volatile` 的作用**：

```java
volatile boolean flag = false;
```

- 读：插入 LoadLoad + LoadStore 屏障 → 从主内存读，禁止读操作重排
- 写：插入 StoreStore + StoreLoad 屏障 → 写直接刷主内存，禁止写操作重排

`volatile` 保证**可见性 + 有序性**，但不保证原子性（`volatile int i; i++` 仍然是竞态）。

**`volatile` vs `atomic` 的区别**：

| | volatile | atomic（AtomicInteger 等）|
|--|---------|------------------------|
| 可见性 | ✅ | ✅ |
| 有序性 | ✅ | ✅ |
| 原子性 | ❌（不保证复合操作原子）| ✅（CAS 实现原子 RMW）|
| 适用场景 | 纯标志位（只写、只读，不做 RMW）| 计数器、并发累加 |

**Python 中的情况**：
- CPython 有 GIL，字节码操作之间不会被其他线程打断，没有 Java 意义上的可见性问题
- 使用 `multiprocessing`（多进程）或 C 扩展时，需要显式同步（`Value + Lock`）
- NumPy 操作释放 GIL 时，需要注意数组并发修改

**考察点**：
1. 为什么多核 CPU 下一个线程的写操作另一个线程不能立刻看到
2. volatile 保证可见性但不保证原子性的典型反例
3. Python GIL 和 Java volatile 的本质区别

**示例答案**：
CPU 有多级缓存，每个核心的 L1/L2 是私有的，线程 A 写了一个变量可能还在自己的 L1 Cache 里没刷到主内存，线程 B 从主内存读到的是旧值——这就是可见性问题。此外 CPU 和编译器还会指令重排（把独立的操作调整顺序提升性能），在单线程下正确，但多线程下可能破坏"先写后读"的假设。内存屏障是解决方案：强制缓存刷新和禁止重排序。Java 的 `volatile` 会在读写两侧插入内存屏障，保证写操作立刻对所有线程可见，读操作拿到最新值。但 `volatile` 不保证原子性：`i++` 是"读-改-写"三步，两个线程都读到旧值后各自 +1 写回，仍然有竞态。`AtomicInteger` 用 CAS 让"读-改-写"原子执行，才是正确的无锁计数器。Python 因为 GIL 的存在，CPython 线程切换只发生在字节码边界，普通 Python 对象的引用计数更新是 GIL 保护的，所以 Python 开发者不太需要关心 volatile 级别的可见性问题——但用 `multiprocessing` 共享内存（`Value`）时，仍然需要显式加锁。

### Q61: 分布式一致性模型有哪些？强一致性 vs 最终一致性 vs 因果一致性的区别？

**题目解析**：一致性模型比 CAP 定理更细粒度，是分布式系统设计中选择数据库、缓存一致性策略的基础，大厂系统设计必考。

**题目讲解**：

**为什么需要一致性模型？**

分布式系统中数据有多个副本，"一致"到什么程度是个权衡：越强一致性，需要的协调越多，延迟越高。

**一致性模型谱系（从强到弱）**：

**① 线性一致性（Linearizability / 强一致性）**

> 系统行为如同只有一份数据，所有操作都有一个全局顺序，每个操作在调用和返回之间的某一时刻"原子发生"。

- 读一定能读到最新的写
- 所有客户端看到的操作顺序完全一致
- **代价**：写入需要等待所有副本确认，或通过全局锁/共识协议协调
- **例子**：Raft/Paxos 集群写入后读取、ZooKeeper 的顺序一致读

```
Client A: write(x=1) ---|--- ack
Client B:            ---|--- read(x) → 一定返回 1
```

**② 顺序一致性（Sequential Consistency）**

> 所有操作看起来有一个全局顺序（但不要求实时），每个线程内部操作顺序保持。

- 比线性一致性弱：操作的全局顺序不需要和实际时间完全对应
- **例子**：CPU 内存模型（TSO/PSO）

**③ 因果一致性（Causal Consistency）**

> 有因果关系（happen-before）的操作，所有节点看到的顺序一致；并发操作可以在不同节点以不同顺序出现。

```
A 发帖（x）→ B 看到 x → B 回帖（y）
因果链：x → y，所有节点都必须先看到 x 再看到 y
与 x 无关的 z 帖子可以在 x 前或后看到（并发）
```

- **例子**：Git（提交依赖关系）、MongoDB causal sessions

**④ 最终一致性（Eventual Consistency）**

> 如果停止更新，数据最终会收敛到一致状态。中间期间不同节点可能看到不同值。

- 最弱的"保证"，但延迟最低，可用性最高
- **例子**：Cassandra、DynamoDB、DNS、Amazon S3

**⑤ 单调读一致性（Monotonic Read）**

> 一个客户端不会读到比之前读到更旧的值（不会"穿越回过去"）。

```
Client 读到 x=2，下次读 x 一定 ≥ 2，不会读到 x=1
```

**⑥ 读己之写（Read-Your-Writes）**

> 写入后，自己能立刻读到刚写的值（但其他客户端可能还看不到）。

```
用户改了密码，下一次自己登录一定用新密码验证
```

**实际系统选型**：

| 系统 | 一致性级别 | 原因 |
|------|-----------|------|
| ZooKeeper | 线性一致性（写）/ 顺序一致性（读）| 分布式协调，不能读脏数据 |
| Cassandra | 最终一致（默认）/ 可调 | 高可用优先，允许暂时不一致 |
| DynamoDB | 最终一致（默认）/ 强一致（可选）| 灵活权衡 |
| MySQL 主从 | 最终一致（从库）/ 强一致（主库）| 读写分离的代价 |
| Redis 主从 | 最终一致（异步复制）| 高性能，接受短暂不一致 |

**考察点**：
1. 线性一致性和顺序一致性的细微区别
2. 因果一致性在社交场景（评论/回复）中的重要性
3. 实际工程中按操作类型分别选择一致性级别

**示例答案**：
CAP 里说的 C 是线性一致性——最强的保证，读到的一定是最新写入的值，所有节点看到完全相同的操作顺序。但现实中大多数 AP 系统采用的是最终一致性：副本异步同步，数据"最终"一致，中间可能读到旧值，Cassandra/DynamoDB 都是这个模型。两者之间有更细粒度的选择：因果一致性只保证有因果关系的操作顺序正确（A 发帖后 B 的回复一定排在 A 帖之后），并发无关的操作不保序，比强一致性代价小得多，社交应用很适合。单调读一致性保证不"穿越回过去"（读到 x=5 后下次一定 ≥5，不会读到 x=3），对用户体验的伤害最小，实现成本低（把请求路由到同一副本即可）。实际工程常见做法：写用强一致性（走主库），读根据业务容忍度分级：订单状态展示走主库（读己之写+线性），商品库存展示走从库（最终一致即可）。

### Q62: Raft 共识算法的工作原理是什么？如何保证即使节点宕机系统也能正确运行？

**题目解析**：共识算法是分布式系统的核心，Raft 是 etcd/TiKV/CockroachDB 的基础，大厂后端面试的高频深度题。

**题目讲解**：

**共识问题是什么？**

> 分布式系统中，让多个节点就某个值达成一致，即使部分节点宕机或网络分区。

这是 Leader 选举、分布式锁、分布式数据库等一切的基础。

**Raft 的三个角色**：

```
Leader   ← 唯一，接收所有写入请求，同步给 Follower
Follower ← 被动接收日志，响应 Leader 心跳
Candidate ← 选举中，发起投票请求
```

**Part 1：Leader 选举**

```
1. 初始状态：所有节点是 Follower
2. Follower 在 150~300ms 随机超时后，没收到 Leader 心跳 → 变成 Candidate
3. Candidate 递增本地 term（任期号），给所有节点发 RequestVote
4. 收到投票请求的节点：
   - 这个 term 还没投过票，且 Candidate 的日志不比自己旧 → 投票
   - 否则拒绝
5. Candidate 收到超过半数（N/2+1）票 → 成为 Leader
6. Leader 定期发心跳（AppendEntries 空消息）维持地位
```

**关键机制 - 随机超时防止分票**：

```
如果所有 Follower 同时超时 → 同时发起选举 → 可能都得不到多数票 → 无限循环
随机超时（150~300ms）让某个节点先超时，大概率在其他节点醒来前就获得多数票
```

**Part 2：日志复制**

```
客户端写入 x=5
    ↓
Leader 写入本地日志（未提交）
    ↓ 并行发 AppendEntries 给所有 Follower
Follower 写入各自日志（未提交），回复 ACK
    ↓
超过半数 Follower 确认（Quorum）
    ↓
Leader 提交（commit）日志，返回客户端"成功"
    ↓ 下一次心跳
告知 Follower 已提交，Follower 也提交
```

**为什么只需要半数确认？**

```
5 个节点：Leader + 2 个 Follower 确认 = 3/5 > 1/2
最多允许 2 个节点宕机，剩余 3 个仍能形成多数
```

**日志的一致性保证**：

Raft 保证：**如果两个节点的日志在某个 index 相同（term 相同），那么这之前所有日志都相同**（AppendEntries 携带前一条日志的 term 和 index 做一致性检查）。

**Part 3：Leader 宕机与重新选举**

```
Leader 宕机 → Follower 超时等不到心跳 → 某个 Follower 发起选举
只有日志最新（term 最大，或 term 相同则 index 最大）的节点才能当选 Leader
→ 确保已提交的日志不会丢失（已提交 = 超过半数节点都有这条日志）
```

**日志压缩（Snapshot）**：

日志无限增长 → 定期 Snapshot（快照当前状态）→ 截断日志，只保留 Snapshot 之后的增量。

**考察点**：
1. 为什么需要随机超时（防止选票分裂）
2. Quorum（半数以上）的意义（最多允许少数节点故障）
3. 已提交的日志为什么不会丢（写入超过半数节点，新 Leader 一定含有它）

**示例答案**：
Raft 把共识问题拆成三个子问题：Leader 选举、日志复制、日志安全。Leader 选举用随机超时解决分票：每个 Follower 随机等 150-300ms，最先超时的那个发起选举，因为随机时间差，它大概率在别人醒来前就获得多数票成为 Leader。日志复制：客户端写请求只走 Leader，Leader 先写本地日志，并行发给所有 Follower，超过半数（Quorum）确认后 commit 并告知客户端成功，再异步通知 Follower 也 commit。Quorum 设计是容错的核心：N 个节点容忍最多 (N-1)/2 个故障，5 节点能容忍 2 个。安全性保证：已提交的日志至少在超过半数节点上存在；新 Leader 选举要求候选者的日志必须是"最新的"（term 和 index 都不落后于投票者），所以新 Leader 一定包含所有已提交的日志，不会丢数据。etcd 和 TiKV 的底层都是 Raft，理解 Raft 就理解了它们为什么能在节点宕机时保持正确性。

### Q63: 分布式数据复制有哪些策略？主从/多主/无主复制分别适合什么场景？

**题目解析**：数据复制策略是分布式存储的基础设计，直接影响系统的一致性、可用性和性能，系统设计面试必考。

**题目讲解**：

**为什么要复制？**
- **高可用**：一台机器宕机，其他副本顶上
- **读扩展**：多副本分担读请求
- **低延迟**：就近副本访问（地理分布）

**策略一：主从复制（Leader-Follower）**

```
写入 → Leader（主库）
读取 → 可以走 Follower（从库）

Leader 把 binlog/WAL 发给 Follower
Follower 异步（默认）或半同步/全同步 回放
```

**同步方式对比**：

| 方式 | 可靠性 | 延迟 | 说明 |
|------|--------|------|------|
| 全同步 | 最高 | 最高 | 所有从库确认才返回 |
| 半同步 | 高 | 中 | 至少一个从库确认 |
| 异步（默认）| 低 | 最低 | 主库不等从库 |

**故障处理**：Leader 宕机 → 需要从 Follower 中选出新 Leader（手动或自动，如 MySQL Sentinel）

**适用场景**：读多写少，允许读从库有短暂延迟（读写分离）。
**例子**：MySQL 主从、PostgreSQL Streaming Replication、Redis Sentinel

---

**策略二：多主复制（Multi-Leader）**

```
多个节点都能接受写入
各自同步给其他主节点
```

**优势**：
- 多数据中心各自写入本地节点，延迟低（不用跨数据中心写入主节点）
- 一个数据中心宕机，其他数据中心继续服务

**核心难题：写冲突**

```
客户端 A 在 DC1 写入 x=1
客户端 B 在 DC2 同时写入 x=2
两个写入都成功，同步后冲突！
```

**冲突解决策略**：
1. **Last-Write-Wins（LWW）**：时间戳大的赢（有丢数据风险，NTP 时钟偏差）
2. **自定义合并**：应用层定义合并逻辑（如 CRDTs、Google Docs OT 算法）
3. **用户解决**：把冲突交给用户（如 Git merge conflict）

**适用场景**：多数据中心、离线优先（如协作文档）。
**例子**：MySQL Group Replication、CouchDB、Google Docs 协作编辑

---

**策略三：无主复制（Leaderless / Dynamo-style）**

```
没有主节点，写入同时发给多个节点
读取也同时读多个节点，取最新值
```

**Quorum 读写**：设 N=3 个副本

```
写：W 个节点写入成功才算成功（W=2）
读：R 个节点响应才算成功（R=2）
W + R > N → 读写集合必然有交集 → 能读到最新写入
```

```
W=2, R=2, N=3：
写入时 2 个节点有最新值
读取时也从 2 个节点读，必然有 1 个交集（有最新值）
```

**读修复（Read Repair）**：
读取时如果发现不同节点数据不一致，顺手把旧的副本更新为最新值。

**适用场景**：高可用优先，能接受最终一致性，写入量大。
**例子**：Cassandra、DynamoDB、Riak

---

**三种策略对比**：

| | 主从复制 | 多主复制 | 无主复制 |
|--|---------|---------|---------|
| 写入点 | 只有 Leader | 多个节点 | 任意节点 |
| 一致性 | 强（同步）/ 最终（异步）| 最终一致 | 最终一致（可调） |
| 冲突处理 | 无冲突 | 需要冲突解决 | 版本向量 |
| 可用性 | Leader 挂了需选举 | 单数据中心挂仍可用 | 少数节点挂仍可用 |
| 典型系统 | MySQL/Redis | CouchDB | Cassandra |

**考察点**：
1. 主从复制读从库可能读到旧数据的场景（写后立刻读）
2. Quorum W+R>N 为什么能保证读到最新值
3. 多主复制的写冲突不可避免，必须设计冲突解决策略

**示例答案**：
主从复制是最常见的方案：写走主库，读可以走从库，适合读多写少场景。代价是主从延迟——异步复制下从库可能落后主库几十毫秒到几秒，"写后立刻读"场景要强制走主库（或等待同步）。多主复制允许在多个数据中心各自接受写入，跨数据中心写延迟从秒级降到毫秒级，但必须处理写冲突——同一数据在两个主节点同时被不同人修改，需要应用层设计合并策略（LWW 丢数据，自定义合并复杂，CRDT 在特定场景下无冲突）。无主复制（Dynamo 模式）用 Quorum 保证 W+R>N，写成功的节点集合和读取的节点集合必然有交集，读一定能拿到最新写入。Cassandra 默认写 W=1，读 R=1（高可用，最终一致），可以设置 W=QUORUM，R=QUORUM（强一致，延迟增加）。选型原则：要强一致性选主从（同步复制），要高可用选无主，要多地域低延迟选多主（接受最终一致）。

### Q64: 什么是分布式时钟？Lamport 时钟和向量时钟如何解决分布式事件排序？

**题目解析**：分布式系统中没有全局时钟，这是分布式调试和因果追踪的核心问题。Lamport 时钟是 distributed systems 领域的经典论文，考察候选人对分布式基础理论的掌握。

**题目讲解**：

**根本问题：没有全局时钟**

```
节点 A 的时间：10:00:00.123
节点 B 的时间：10:00:00.119（时钟漂移 4ms）
```

- NTP 同步精度一般在毫秒级，无法精确排序微秒级的事件
- 即使时钟完美同步，网络延迟也不确定，"先发生"和"先到达"不一样
- **结论**：物理时间无法可靠确定分布式事件的全局顺序

**Happened-Before 关系（先发生关系）**：

Lamport 1978 年定义：事件 a → b（a 先发生于 b），当且仅当：
1. a 和 b 在同一进程，a 排在 b 前面
2. a 是发送消息，b 是接收同一条消息
3. 存在 c 使得 a → c → b（传递性）

**Lamport 逻辑时钟**：

```python
clock = 0

# 本地事件发生
def local_event():
    global clock
    clock += 1

# 发送消息
def send_message(msg):
    global clock
    clock += 1
    msg.timestamp = clock
    network.send(msg)

# 接收消息
def receive_message(msg):
    global clock
    clock = max(clock, msg.timestamp) + 1   # 关键规则
```

**Lamport 时钟的保证**：

```
a → b  ⟹  clock(a) < clock(b)      ✅ 必然成立
clock(a) < clock(b)  ⟹  a → b      ❌ 不一定成立（反向不成立）
```

> 并发事件的 Lamport 时钟可以任意大小，无法区分"因果先后"和"纯巧合顺序"。

**向量时钟（Vector Clock）**：

每个进程维护一个向量 `[t1, t2, t3]`（每个进程各一个计数器）：

```python
# N 个进程，进程 i 的向量时钟
vc = [0, 0, 0]   # [进程0时钟, 进程1时钟, 进程2时钟]

# 进程 i 的本地事件
vc[i] += 1

# 进程 i 发送消息
vc[i] += 1
send(msg, vc.copy())

# 进程 i 接收来自 j 的消息（消息携带 vc_msg）
vc[k] = max(vc[k], vc_msg[k]) for each k
vc[i] += 1
```

**向量时钟的判断能力**：

```
事件 A 的向量时钟：[2, 1, 0]
事件 B 的向量时钟：[3, 2, 0]

A ≤ B（每一维都 ≤）→ A → B（A 先发生于 B）

事件 C 的向量时钟：[2, 0, 1]
A vs C：A=[2,1,0], C=[2,0,1]，无法比较（A[1]=1 > C[1]=0，但 A[2]=0 < C[2]=1）
→ A 和 C 是并发的（concurrent）
```

**向量时钟的意义**：

```
a → b  ⟺  vc(a) < vc(b)     ✅ 充要条件
a || b（并发）  ⟺  vc 不可比较   ✅
```

> 这就是 Git 合并的底层：判断两个提交是否有因果关系，没有就是"并发分叉"，需要 merge。

**实际应用**：

| 应用 | 使用场景 |
|------|---------|
| Amazon DynamoDB | 向量时钟检测写冲突（每个副本一个计数器）|
| Riak | 向量时钟判断哪个版本更新 |
| Git | DAG（有向无环图）判断提交因果关系 |
| 分布式 Tracing | Span 的 parent-child 关系（因果链路）|

**考察点**：
1. Lamport 时钟的单向性（→ 推出时钟，但时钟推不出 →）
2. 向量时钟如何区分"因果"和"并发"
3. 向量时钟的空间开销（随进程数线性增长，大集群要压缩）

**示例答案**：
分布式系统没有全局时钟，NTP 也只能同步到毫秒级，对于微服务间的事件排序完全不够用。Lamport 逻辑时钟用一个规则建立因果关系：本地事件 +1，接收消息时取 max(本地, 消息时钟)+1。这样如果 A 先发生于 B，A 的逻辑时钟一定小于 B 的——但反过来不成立，时钟小不代表先发生，只代表可能先发生。向量时钟解决了这个问题：每个进程一个计数器，向量的每一维都记录"这个进程直接或间接知道对方做了多少事"。比较两个向量时钟可以准确判断：一个每维都小于另一个 → 有因果关系；有维互相大小交叉 → 并发，谁也没先发生。DynamoDB 的版本冲突检测就是向量时钟：写入 A 的版本是 [1,0]，写入 B 的版本是 [0,1]，不可比较 → 并发写冲突，需要用户选择或应用层合并。分布式链路追踪（OpenTelemetry）里 Span 的 parent-child 也是因果链，虽然实现上用 traceID+spanID 而非完整向量时钟，但思想相通。
